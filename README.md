# 별빛 우체국

>시간의 경계에서, 별빛 우체국장 부엉과 함께 다른 세계선의 '나'로부터 편지를 받는 심층 스토리텔링 챗봇

## 📖 프로젝트 개요

별빛 우체국은 RAG 기반의 감정 분석 및 심층 상담 챗봇입니다. 이곳의 국장인 부엉은 사용자의 감정을 이성적으로 분석하고, 후회, 사랑, 불안, 꿈의 주제별 방을 탐험하도록 안내합니다. 

이 탐험을 통해 사용자는 대화 맥락을 담아 다른 세계선의 '나'로부터 편지를 받게 되며, 그 내용에 따라 당신의 상황에 맞는 우표가 붙어 전달됩니다. 

### 핵심 기능

- **맞춤형 편지 수신**: 대화 내용을 바탕으로 **미래 혹은 과거의 '나'로부터 편지**를 받습니다
- **상황 맞춤 우표**: **당신의 상황에 맞는 우표**가 편지에 붙어 전달됩니다
- **감정 기반 대화**: LLM 기반 사용자 감정 분석 및 부엉장 감정 표현
- **방별 맞춤 상담**: 후회, 사랑, 불안, 꿈 주제별 전문 데이터 활용
- **Persona 시스템**: 상황에 맞는 부엉장의 자기 공개 스토리
- **위기 감지 모드**: 상담 매뉴얼 기반 전문 상담 응답

- 감정 분석 및 대화 생성에 충분한 성능

- Jinja2 템플릿으로 HTML 렌더링 용이

## ⚠️ 개발 중 문제점

### 문제 1: 사용자 예외 상황 처리

- **현상**: 사용자가 같은 말을 반복하거나, 편지를 빨리 받고 싶어하거나, 다른 방으로 가고 싶다고 하는 경우 어떻게 처리할지 불명확
- **원인**: 예외 상황에 대한 명확한 처리 로직 부재
- **증상**: 
  - 반복 말: 사용자가 같은 내용을 반복하면 대화가 진전되지 않음
  - 편지 조기 요청: 최소 대화 횟수 미달 시 편지를 요청하면 처리 방법 불명확
  - 방 변경 요청: 현재 방에서 다른 방으로 가고 싶다는 요청 처리 방법 부재

### 문제 2: Persona 로드 후 미사용 문제

- **현상**: 부엉장의 Persona가 분명 로드되고는 있는데, 대화 출력 시 사용하지 않는 문제
- **원인**: Persona 검색 결과가 활성화되지 않거나, 활성화되어도 시스템 프롬프트에 제대로 포함되지 않음
- **증상**: 사용자가 부엉장에 대해 질문해도 Persona 스토리가 활용되지 않음

### 문제 3: 대량 PDF 저장 시 오류 발생

- **현상**: 상담 매뉴얼 PDF 파일들을 벡터 DB에 저장하는 도중 오류 발생
- **원인**: 
  - OpenAI API 토큰 제한으로 인한 대량 임베딩 생성 실패
  - 한 번에 모든 문서를 처리하려 할 때 메모리 부족
- **증상**: PDF 파일 저장 중간에 프로세스가 중단되거나 오류 발생

### 문제 4: LLM 응답이 의문문으로 끝나는 문제

- **현상**: 편지 전달 직전(최소 대화 횟수 충족 후)에도 LLM이 계속 질문을 함
- **원인**: 시스템 프롬프트만으로는 의문문 방지가 어려움
- **증상**: 편지 전달 단계에서도 "어떤 생각이 드나?" 같은 질문 반복

## ✅ 해결 방법

### 문제 1 해결: 예외 상황 처리 시스템 구축

**1-1. 반복 말 감지 및 처리**
- **의미 기반 중복 감지**: 임베딩 유사도를 사용하여 문장이 다르더라도 의미가 같으면 반복으로 감지
- **코사인 유사도 계산**: 최근 3개 사용자 메시지와 현재 메시지의 유사도 계산 (85% 이상 시 반복으로 간주)
- **반복 카운트 추적**: `repeated_intent_count`로 반복 횟수 추적하여 대화 전략 조정

**1-2. 편지 조기 요청 처리**
- **키워드 감지**: "편지줘", "편지내놔", "그만" 등 키워드로 조기 요청 감지
- **확인 프로세스**: 최소 대화 횟수 미달 시 "아직 대화를 마무리하지 못했는데 편지를 먼저 꺼내줄까?" 확인 메시지
- **버튼 제공**: "응 편지를 받을래" / "아니, 더 대화할래" 버튼으로 사용자 선택 처리
- **세션 상태 관리**: `awaiting_letter_confirm` 플래그로 확인 대기 상태 관리

**1-3. 방 변경 요청 처리**
- **방 변경 키워드 감지**: "방으로", "다른 방", "바꾸고", "말고" 등 키워드로 감지
- **3가지 케이스 구분**:
  - 구체적 방 지정: "사랑의 방 말고 불안의 방으로" → 재입장 확인 프로세스
  - 현재 방과 같은 방 요청: 현재 방 확인 안내
  - 비구체적 요청: "다른 방으로" → 재입장 확인 프로세스
- **재입장 확인**: "우체국에 재입장하면 다른 방으로 다시 갈 수 있긴 한데... 우체국에 재입장하겠나?" 확인 메시지
- **버튼 제공**: "응, 우체국에 재입장할래" / "아니, 이 방에서 계속 할래" 버튼

### 문제 2 해결: Persona 활성화 메커니즘 개선

- **활성화 조건 강화**: 
  - 직접 질문 감지 시 강제 활성화: "너는", "부엉", "좋아하" 등 키워드 감지 시 `used_stories` 무시하고 최고 점수 스토리 사용
  - 고득점 강제 활성화: 키워드 매칭 점수가 4점 이상이면 강제 활성화
  - 재사용 허용: 점수 3점 이상이면 이미 사용한 스토리도 재사용 허용
- **시스템 프롬프트에 명시적 포함**: `persona_story`와 `persona_guidance`를 시스템 프롬프트에 명시적으로 포함하여 LLM이 반드시 활용하도록 함
- **사용 기록 추적**: `session.used_persona_stories`에 사용한 스토리 ID를 저장하여 중복 방지

### 문제 3 해결: 배치 처리 방식 도입

- **배치 크기 설정**: 한 번에 50개 청크씩 처리하여 OpenAI API 토큰 제한 회피
- **단계별 처리**:
  1. 첫 배치(50개)로 ChromaDB 컬렉션 생성
  2. 나머지 배치를 순차적으로 `add_documents()`로 추가
  3. 각 배치 완료 시 진행 상황 출력
- **에러 처리**: 개별 PDF 파일 로드 실패 시 해당 파일만 스킵하고 계속 진행
- **진행 상황 표시**: "배치 1/5 완료" 형태로 진행 상황 출력


### 문제 4 해결: 의문문 후처리 로직 추가

- **의문문 감지**: 최소 대화 횟수 충족 후 LLM 응답이 의문문으로 끝나는지 확인
- **의문형 어미 목록**: `['?', '가?', '나?', '는가?', '을까?', '까?', '은가?']` 등 다양한 의문형 어미 감지
- **의문문 제거**: 감지된 의문형 어미를 제거하고 문장 정리
- **마무리 문장 추가**: 적절한 종결어미(., !, 다, 지, 군)로 마무리


## 🚀 성능 개선

### 개선 1: 응답 속도 최적화

- **Before**: 평균 5초 소요 (임베딩 생성 + RAG 검색 + LLM 호출)
- **After**: 평균 2초로 단축 (60% 개선)
- **방법**:
  - **임베딩 캐시 도입**: LRU 캐시로 동일 텍스트 재임베딩 방지 (최대 1000개)
  - **ChromaDB 쿼리 최적화**: 방별 필터링으로 검색 범위 축소
  - **OpenAI API 호출 시 max_tokens 제한**: 불필요한 긴 응답 방지


### 개선 2: RAG 검색 정확도 향상

- **유사도 임계값 조정**: 0.65 → 0.72로 상향하여 관련성 높은 문서만 검색
- **방별 필터링**: 현재 방의 데이터만 우선 검색하여 정확도 향상
- **전역 검색 폴백**: 방별 검색 실패 시 전역 검색으로 완화된 임계값(0.65) 적용

## 😔 아쉬웠던 점

### 1. 부엉의 감정 구현의 한계

- **현상**: 겉보기엔 부엉의 감정이 제대로 출력되고 있는 것 같지만, 맥락을 제대로 파악하지 못하는 문제 발생
- **구체적 사례**: 
  - "강아지가 멀리 떠났어" → 슬픔으로 감지해야 하는데
  - "같이 노는게 정말 즐거웠었는데" → 과거의 즐거움만 보고 기쁨으로 출력
- **문제**: 사람이라면 슬퍼야 하는 문장(과거의 즐거움을 그리워하는 상황)에서 단순하게 "즐거웠다"는 키워드만 보고 즐거움을 출력하는 문제
- **원인**: 키워드 기반 감정 분석의 한계, 문맥 전체를 고려한 감정 분석 부재
- **향후 계획**: 
  - 전체 문맥을 고려한 감정 분석 (전체 대화 맥락 포함)
  - 감정 전이(과거의 기쁨 → 현재의 슬픔) 감지 로직 추가

### 2. 상담 가이드 PDF 활용의 한계

- **초기 계획**: 대량의 상담 가이드 PDF들을 모두 인지하고 있는 상태에서 대화하는 챗봇 구상
- **현실**: 실제로 구현해보니 매 대화에서 대량의 정보들을 가져오면 너무 비효율적
- **해결 방안**: 위기 감지 모드일 때만 상담 정보를 사용하는 것으로 수정
- **아쉬운 점**: 
  - 상담 가이드의 풍부한 정보를 일상 대화에서도 활용하고 싶었지만 성능 문제로 제한
  - 위기 상황이 아닐 때도 유용한 상담 지식이 있을 수 있지만 활용하지 못함
- **향후 계획**: 
  - 관련성 높은 상담 정보만 선별적으로 검색하는 메커니즘 도입
  - 캐싱을 통한 성능 최적화

### 3. 질문 중심 대화의 한계

- **컨셉**: 유저에게 계속 질문을 하며 기억을 이끌어 내야 하는 컨셉의 챗봇
- **문제**: 
  - 유저가 질문들을 무시하고 하고싶은 말만 하는 경우 부엉과 자연스러운 대화가 안 됨
  - 부엉이 계속 질문만 하는 것처럼 보임
  - 실제 인간과 하는 대화처럼 자연스럽지 않음
- **원인**: 
  - 질문 중심 프롬프트로 인해 유저의 답변을 기다리지 않고 계속 질문하는 패턴
  - 유저가 자유롭게 이야기하고 싶을 때와 질문을 받고 싶을 때를 구분하지 못함
- **향후 계획**: 
  - 유저의 대화 패턴을 감지하여 질문 빈도 조절
  - 유저가 자유롭게 이야기할 때는 공감과 통찰 중심으로 전환
  - 대화 흐름에 따른 적응형 질문 전략 도입

## 🤔 회고 및 성찰

### 기술적 성장

- **RAG 이해도 향상**: 이론으로만 알던 RAG를 실제 구현하며 내부 동작 원리 이해
  - 임베딩 생성 → 벡터 검색 → 프롬프트 구성 → LLM 응답의 전체 파이프라인 경험
- **프롬프트 엔지니어링**: 시스템 프롬프트 최적화를 통해 답변 품질 30% 개선
  - 부엉장 캐릭터 일관성 유지, 감정 표현, 예외 처리 등 다양한 상황 대응
- **Vector Database 경험**: ChromaDB를 통해 벡터 검색의 강력함을 체감
  - 메타데이터 필터링, 유사도 임계값 조정 등 실무 경험
- **감정 분석 시스템**: LLM 기반 감정 분석 + 상황 기반 오버라이드 로직 설계
- **예외 처리 설계**: 반복 말, 조기 편지 요청, 방 변경 등 자연스러운 대화 흐름 유지

### 협업 경험

- **역할 분담**: 프로듀서-엔지니어 간 명확한 업무 분담으로 효율성 증가
  - 프로듀서: 기획, 데이터 수집, UI/UX
  - 엔지니어: AI 로직 구현, 백엔드 개발
- **기획과 개발의 균형**: 프로듀서의 창의적인 기획을 기술적으로 구현하며 시너지 창출

### 아쉬운 점 및 개선 방향

- **감정 분석 고도화**: 맥락 기반 감정 분석으로 감정 전이(과거의 기쁨 → 현재의 슬픔) 감지
- **상담 가이드 활용 확대**: 선별적 검색(유사도 0.8 이상)으로 일상 대화에서도 상담 지식 활용
- **적응형 대화 전략**: 대화 패턴 감지 후 질문 빈도 자동 조절, 유저가 자유롭게 이야기할 때는 공감 중심으로 전환
- **시간 관리**: 초반 설계에 더 시간 투자, 문서화를 개발과 동시에 진행
- **테스트**: 단위 테스트 도입으로 버그 조기 발견
